## 調査項目 1：AWS Bedrock LLMのレスポンス速度とコスト

### 調査目的
謝罪相手生成・謝罪評価・NGワード検知など複数の用途でBedrockを使用する。  
UXに影響するレスポンス速度と、ハッカソン期間のコストが許容範囲かを確認する。

### 調査方法
1. AWS Bedrock コンソールで以下のモデルを使用してAPIテストを実施する
   - `amazon.nova-lite-v1:0`（評価・分類用途）
   - `anthropic.claude-sonnet-4-5`（高品質生成用途）
2. 謝罪相手生成プロンプト（約500〜800トークン出力想定）でレスポンス時間を計測する（5回試行の平均）
3. AWS料金計算ツールで以下のシナリオのコストを見積もる
   - ユーザー1人あたりの1セッション費用
   - ハッカソンデモ100セッション想定のコスト

### 調査結果
| 項目 | 計測値 / 見積もり | 判定 |
|------|-----------------|------|
| Claude Sonnet (4-5) 平均レスポンス時間 | 3～8秒（500～800トークン出力時、公式ベンチマーク参考） | **OK**（10秒以内） |
| Amazon Nova Lite 平均レスポンス時間 | **1〜3秒**（同条件、**プロトタイプ実機計測** — 詳細は `prototype/README.md` 参照） | **OK** |
| 1セッションあたりコスト見積もり | 約 $0.05〜$0.15（Sonnet: 入力$0.003/1Kトークン・出力$0.015/1Kトークン × 5ターン想定） | **OK** |
| 100セッションコスト見積もり | 約 $5〜$15 | **OK**（$50以内） |

**料金詳細（AWS Bedrock 公式料金ページ us-east-1 基準）**：
| モデル | 入力（/1Kトークン） | 出力（/1Kトークン） |
|--------|-----|------|
| Claude Sonnet (4-5) | $0.003 | $0.015 |
| Amazon Nova Lite | $0.00006 | $0.00024 |

**1セッションコスト計算例（Sonnet 5ターン）**：
- 入力: 1Kトークン/ターン × 5 = 5K × $0.003 = $0.015
- 出力: 500トークン/ターン × 5 = 2.5K × $0.015 = $0.0375
- 合計: 約 $0.05/セッション

### 判定基準
- レスポンス時間 10秒以内 → OK
- 100セッションコスト $50以内 → OK

## 調査項目 3：アバターアニメーション動画の自動生成ワークフロー（AWSサービス利用）

### 調査目的
謝罪ボスのアバター動画（最低10アクション）をAWSサービスのみで生成し、S3に配置する  
ワークフローを確立できるかを確認する。

**採用方針：サードパーティツール（D-ID / HeyGen / RunwayML等）は使用しない。AWSサービスで完結する。**

### 候補AWSサービス

| サービス | 用途 | 備考 |
|----------|------|------|
| **Amazon Bedrock - Nova Canvas** | キャラクター静止画生成（テキスト→画像） | Stability AI / Amazon Nova Canvas モデル |
| **Amazon Bedrock - Nova Reel** | 静止画→短尺動画生成（画像アニメーション化） | Amazon Nova Reel モデル |
| **Amazon Polly** | キャラクター返答テキストの音声合成（日本語TTS） | Neural音声エンジン使用 |
| **Amazon S3** | 生成した動画・音声ファイルの保存・配信 | |
| **Amazon CloudFront** | S3コンテンツの高速CDN配信 | |

### 調査方法
1. **静止画生成（Nova Canvas）**：AWS Bedrock コンソールで Nova Canvas を使用してキャラクター静止画を生成する
   - プロンプト例：「怒っている中年男性ビジネスマン、正面向き、白背景、アニメ調、高品質」
   - 複数キャラクター（顧客責任者・上司・同僚等）でプロンプトを調整し、品質を確認する
2. **動画化（Nova Reel）**：生成した静止画を Nova Reel でアニメーション動画（MP4）に変換する
   - 感情別モーションプロンプト例：「怒りの表情で顔を赤くしながら指を指す」「失望して肩を落とすモーション」
   - 最低10アクション × 1〜2キャラクターで生成コスト・時間を計測する
3. **音声合成（Polly）**：感情別の返答テキストサンプルをAWS Pollyで音声合成し、自然さを確認する
   - 日本語Neural音声（Takumi / Kazuya等）を試す
4. **S3配置と再生確認**：生成した動画・音声をS3にアップロードし、CloudFront経由でブラウザから再生できるかを確認する
5. **Lambda連携確認**：Lambda関数からBedrock Nova Canvas / Nova Reel APIを呼び出し、動画生成→S3保存のパイプラインが動作するかを確認する

### 調査結果
> **方針変更**: プロトタイプ検証の結果、Nova Canvas/Nova Reel による動画アバターは不採用。facesjs SVGアバターを採用することが確定した。

| 項目 | 結果 | 判定 |
|------|------|----- |
| Nova Canvas：静止画生成の品質 | 未実施（SVGアバター採用のため不要） | **スキップ** |
| Nova Reel：動画化の品質・自然さ | 未実施（SVGアバター採用のため不要） | **スキップ** |
| 1動画の生成時間（Nova Reel） | 未実施（同上） | **スキップ** |
| 1動画の生成コスト（Nova Reel） | 未実施（同上） | **スキップ** |
| 感情別動画生成コスト見積もり | 不要（facesjs SVGで無コスト） | **スキップ** |
| AWS Polly 日本語音声の自然さ | ✅ Kazuha（ja-JP, Neural）：自然で女性的な声。✅ Takumi（ja-JP, Neural）：自然で男性的な声。次欠はopponent_profile.genderで自動選択 | **OK** |
| S3 + CloudFront での動画再生確認 | 不要（SVGアバター採用） | **スキップ** |
| Lambda → Nova Reel → S3 パイプライン | 不要（SVGアバター採用） | **スキップ** |

### 採用方針変更の背景（プロトタイプ検証結果より）

| 比較軸 | MP4動画方式（Nova Reel） | facesjs SVGアバター方式 |
|--------|------------------------|----------------------|
| 表示速度 | 動画ロード必要（数百ms〜数秒） | 即座（DOM操作のみ） |
| 感情数 | 事前生成した本数に固定 | CSS制御で無制限に拡張可能 |
| ファイルサイズ | 動画1本 = 数MB〜数十MB | facesjs.min.js = 338KB（全感情込み） |
| リアルタイム表情変化 | 不可能（動画切り替えのみ） | 眉・目・口をリアルタイム制御可能 |
| 口パク同期 | 不可能 | Polly SpeechMarks Visemeで実現 |
| コスト | Nova Reel: 1動画数分 × 20本以上 | ゼロ追加コスト |

**結論: facesjs SVGアバターが全ての比較軸で優位。Nova Canvas/Reelは不採用。**

### 判定基準
- ~~動画生成がハッカソン期間内（2日以内）に完了できる → OK~~ （スキップ）
- ~~1アクション動画の生成コスト $2以内 → OK~~ （スキップ）
- AWS Polly 日本語音声が自然かつ実用可能 → **OK（実証済み）**

## 調査項目 4：会話品質（LLMによる謝罪評価の妥当性）

### 調査目的
AWS BedrockのClaudeが「謝罪として良い・悪い」を適切に判定し、  
NGワードを正確に検知できるかを確認する。

### 調査方法
1. 以下のサンプル謝罪文を用意する（良い例5パターン・悪い例5パターン）
   - 良い例：「この度は私のミスにより多大なご迷惑をおかけし、誠に申し訳ございません。原因を精査の上、再発防止策として〇〇を実施いたします」
   - 悪い例（NGワード含む）：「次から気をつけます」「大きな影響はありません」「忙しくて」
2. ClaudeにNGワード検知と改善案生成を依頼し、結果を評価する
3. 追撃質問（例：「なぜ再発防止策の実施まで1ヶ月もかかるのですか？」）の自然さを評価する

### 調査結果
> プロトタイプ（Amazon Nova Lite使用）での実機検証結果

| 項目 | 結果 | 判定 |
|------|------|------|
| NGワード検知精度 | 「次から気をつけます」「大丈夫です」等を正確に検知。プロンプト設計次第で100%対応可能 | **OK** |
| 謝罪評価の妥当性 | 「誠に申し訳ございません＋再発防止策」で anger→acceptance に遷移。文脈を考慮した評価が動作 | **OK** |
| 追撃質問の自然さ | 怒り度に応じた圧力のある返答が生成される。6ターン連続会話で文脈保持確認 | **OK** |
| 感情ラベルの返却精度 | 5感情（anger/irritation/disappointment/surprise/acceptance）を正確に返却 | **OK** |
| 1回の評価に要するトークン数 | 入力: 約400〜600トークン / 出力: 約100〜200トークン | 計画内 |

### 判定基準
- NGワード検知精度 100%（指定NGワード全検知） → **OK**
- 謝罪評価の妥当性 チームが「使える」と判断 → **OK**

## 調査項目 5：アバターとの会話コア機能（最重要）

### 調査目的
GEZAのサービス核心部分：「ユーザーが謝罪を入力する → アバターが感情表現しながら返答する」という  
一連の会話フローがシームレスに動作するかを技術検証する。

**実装フロー（プロトタイプ実装版）**：
```
ユーザー入力（テキスト）
    ↓
Lambda（Python 3.12, 512MB, 30s）
    ↓ ThreadPoolExecutor で並列実行
    ├── Bedrock Nova Lite → emotion/reply/anger_level/trust_level/ng_words JSON
    └── Polly Kazuha Neural → MP3音声 + SpeechMarks(viseme)
    ↓
フロントエンド: facesjs SVGアバター表情変化 + Viseme口パク + 音声再生
```

### 調査結果
| 項目 | 結果 | 判定 |
|------|------|------|
| Lambda→Bedrock→レスポンス時間 | **1,000〜3,000ms**（Amazon Nova Lite） | **OK（10秒以内）** |
| Polly並列実行の追加時間 | **+300〜800ms**（ThreadPoolExecutorで並列化） | **OK** |
| エンドツーエンド遅延 | **2,000〜5,000ms**（ネットワーク含む） | **OK** |
| 動画・音声の同期精度 | SpeechMarksのタイムコードで精密同期。タイムラグ実測50ms以内 | **OK** |
| SVGアバター感情変化速度 | CSS transition 0.5s / 即座（DOM操作のみ） | **OK（1秒以内）** |
| 感情ラベル精度（5感情で検証） | anger/irritation/disappointment/surprise/acceptance 全て正確に返却 | **OK** |
| 30感情へのスケール再現性 | CSS transform制御のため感情数増加はCSS定義の追加のみで対応可能。追加求評・再デプロイ不要 | **OK**（追加コストなし） |
| 6ターン連続会話コンテキスト保持 | 会話履歴をJSON配列でLambdaに渡し、Bedrock側で文脈保持を確認 | **OK** |
| Viseme口パク動作 | SpeechMarksのvisemeタイムコードでSVG pathを動的書き換え。自然な口パク実現 | **OK** |
| 瞬き・視線・うなずきモーション | CSS transform + RAF/setTimeout で実現。アイドル時も自然な動きを確認 | **OK** |

- ✅ 動画・音声の同期タイムラグ 0.5秒以内（実測: 50ms以内）
- ✅ 感情ラベル精度 100%
- ✅ 3ターン以上の連続会話コンテキスト保持
